# Release artifacts cross-compile from a single Linux host

Release builds produce all six supported triples — `x86_64-linux-musl`, `x86_64-linux-gnu`, `aarch64-linux-musl`, `aarch64-macos`, `x86_64-windows-gnu`, `aarch64-windows-gnu` — from one `ubuntu-latest` GitHub Actions runner via a `zig build release` step that loops `b.standardTargetOptions`-style cross-compiles. Zig 0.16.0 is pinned exactly (in `.mise.toml` and the workflow's `setup-zig`) so the same source revision produces byte-identical outputs across rebuilds; combined with `ReleaseSafe`, stripped debug info, and a `-Drelease-version=` build option for the embedded version string, this gives Level-2 reproducibility (same recipe = same bytes across hosts, but not the Debian-grade third-party reproducibility that would require `SOURCE_DATE_EPOCH` and path/timestamp normalization). The native-per-platform alternative — one CI job per OS each running its own `zig build` — was rejected because it makes host the implicit variable in the artifact and loses the bit-identical-rebuild promise without buying anything `tk` needs: there is no platform-specific toolchain step (no Apple codesigning/notarization, no MSVC ABI link, no Windows resource compiler), and Zig's cross-compilation natively supplies libc shims and headers for every target. Linkage policy is fixed per triple: musl builds are fully static (the niche is containers, Alpine, and minimal CI environments — not "supports old distros"); glibc builds are dynamic with a `2.28` floor (RHEL 8 / Debian 10 / Ubuntu 18.04 — costless portability insurance); macOS uses dynamic `libSystem` with `-mmacos-version-min=11.0` because Apple forbids statically linking it; Windows uses `-static-libgcc` with dynamic `msvcrt` so the shipped `tk.exe` is a single file without a `libgcc_s_seh-1.dll` neighbor. Smoke verification then runs per-platform on native GitHub runners (`ubuntu-latest`, `ubuntu-24.04-arm`, `macos-14`, `windows-latest`, and `windows-11-arm` for aarch64-windows) by downloading the cross-compiled artifact and running a minimal `tk init / add / list` scenario against it — this tests *the shipped bytes*, not a fresh native rebuild. The `aarch64-windows-gnu` smoke job is `continue-on-error: true` because `windows-11-arm` is in GitHub Actions preview; the release-publish step gates artifact upload on smoke success, so that triple is best-effort per release (a failed or unavailable smoke means the artifact is simply absent from that GitHub Release rather than blocking publication). Once `windows-11-arm` graduates from preview and has been green for several releases, the `continue-on-error` flag drops and aarch64-windows-gnu becomes required.
