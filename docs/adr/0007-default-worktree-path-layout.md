# Default worktree path is `<parent-of-main-toplevel>/<repo-basename>.<id>-<slug>`

> **Superseded by ADR-0022.** `tk worktree start` — the only consumer of this
> layout — is removed; worktree creation is handed to the harness/`git
> worktree`. This ADR is retained for historical context.

`tk worktree start <id>` defaults the new **Workspace**'s filesystem path to a flat sibling of the main worktree, prefixed by the repo basename and suffixed by the **Ticket** or **Epic** **Display ID** plus a sanitized title slug capped at 30 characters at the last `-` boundary that fits. For a repository at `~/dev/foo/bar` and a Ticket `proj-1` titled "Fix login bug", the worktree lands at `~/dev/foo/bar.proj-1-fix-login-bug`. Three layouts were considered: flat siblings with no namespace (`<parent>/<id>-<slug>`), a dot-prefixed subdirectory (`<parent>/<repo>.worktrees/<id>-<slug>`), and the chosen `<parent>/<repo>.<id>-<slug>`. The chosen layout matches a manual flat-sibling pattern that already works well with zoxide-style fuzzy-jump tools (`z bar.proj-1`); the repo-basename prefix avoids cross-repo path collisions on the same parent directory; the Display ID anchors identity so `tk worktree` and the filesystem agree; and the truncated title slug carries enough memory hook that a human can recall *what* a path is for without consulting `tk worktree`. The 30-character slug cap was set after sampling 2988 branches from a real monorepo (slug length: median 21, p75 27, p90 34, p95 40) — 30 lets typical 3-word titles pass through untruncated while keeping outlier titles short enough for tab completion. The full branch slug retains a more generous 40-character cap because branches are read in `git branch -a` listings rather than typed. The explicit `[path]` positional remains the escape hatch for users who prefer a single-word manual slug like `bar.feature-x`; configurable layout (per-repo or per-user templates) is deferred from v1.
